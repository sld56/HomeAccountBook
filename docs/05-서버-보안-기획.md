## 1. 전체 아키텍처

```
[가족 브라우저]
    │  HTTPS (Cloudflare auto TLS, HSTS)
    ▼
[Cloudflare Edge — gagyebu.yourdomain.com]
    │  outbound tunnel (인바운드 포트 0개)
    ▼
[미니 PC — Docker Compose]
  ├── cloudflared          (터널 데몬, outbound only)
  ├── kong                 (게이트웨이 / 라우팅)
  ├── gotrue               (Auth — 매직링크 / OTP / 세션)
  ├── postgrest            (자동 REST API, RLS로 보호)
  ├── realtime             (WebSocket 변경 알림, 선택)
  ├── edge-functions       (초대 검증 등 신뢰 작업, Deno 런타임)
  ├── studio               (관리 UI, LAN 또는 Tailscale로만 접근)
  └── postgres             (도메인 테이블 + RLS 정책 + 감사 로그)
              │
              ▼
```

별도 백엔드 서버 코드 없음. **격리는 DB의 RLS가 강제**, 외부 노출은 Cloudflare Tunnel만.

## 2. 데이터 모델

### 2.1 인증/조직 테이블

```sql
-- 가족(워크스페이스) — 모든 도메인 데이터의 루트
CREATE TABLE households (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name        text NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  created_by  uuid NOT NULL REFERENCES auth.users(id)
);

-- 가족 ↔ 사용자 매핑 (권한)
CREATE TABLE household_members (
  household_id uuid NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  user_id      uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role         text NOT NULL CHECK (role IN ('owner','member')),
  display_name text NOT NULL,
  short        text NOT NULL,                -- 아바타 한 글자
  color_key    text NOT NULL,                -- 'appa' | 'eomma' | 'deahyun' | 'jiwon'
  joined_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (household_id, user_id)
);

-- 초대 (24h TTL, 1회용)
CREATE TABLE invitations (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id  uuid NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  invited_by    uuid NOT NULL REFERENCES auth.users(id),
  email         citext,                                -- 매직링크 발송 대상
  token_hash    bytea NOT NULL,                        -- sha256(raw_token), raw는 URL에만
  role          text NOT NULL CHECK (role IN ('owner','member')) DEFAULT 'member',
  created_at    timestamptz NOT NULL DEFAULT now(),
  expires_at    timestamptz NOT NULL,
  consumed_at   timestamptz,
  consumed_by   uuid REFERENCES auth.users(id),
  revoked_at    timestamptz,
  CHECK (expires_at <= created_at + interval '24 hours')
);

CREATE INDEX invitations_token ON invitations (token_hash) WHERE consumed_at IS NULL AND revoked_at IS NULL;
```

### 2.2 도메인 테이블 (모두 household_id 필수)

```sql
CREATE TABLE accounts (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id  uuid NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  label, type, bank, balance, color, "limit"
  created_by, updated_by, created_at, updated_at
);

CREATE TABLE transactions (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id  uuid NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  date date NOT NULL, kind text CHECK (kind IN ('in','out')),
  amount bigint NOT NULL CHECK (amount > 0),
  cat text NOT NULL, title text NOT NULL, memo text,
  member_id uuid REFERENCES household_members(user_id),  -- 누가
  account_id uuid REFERENCES accounts(id),
  created_by uuid NOT NULL REFERENCES auth.users(id),
  updated_by uuid REFERENCES auth.users(id),
  created_at, updated_at
);

CREATE INDEX transactions_household_date ON transactions (household_id, date DESC);
CREATE INDEX transactions_household_member ON transactions (household_id, member_id);

-- budgets, goals, upcoming도 동일 패턴 (household_id + FK + 인덱스)
```

### 2.3 감사 로그 (append-only)

```sql
CREATE TABLE audit_log (
  id            bigserial PRIMARY KEY,
  household_id  uuid REFERENCES households(id),
  actor_user_id uuid REFERENCES auth.users(id),
  action        text NOT NULL,        -- 'tx.create', 'tx.update', 'invite.consume', 'member.remove'
  target_table  text,
  target_id     text,
  diff          jsonb,                -- {before, after} — 금액·이름은 마스킹
  ip_hash       bytea,                -- sha256(ip + daily_salt) — Cloudflare가 전달
  user_agent    text,
  created_at    timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX audit_household_time ON audit_log (household_id, created_at DESC);
-- INSERT만 허용, UPDATE/DELETE 정책 없음
```

### 2.4 user_id ↔ member 매핑 변경

기존 클라이언트의 `Member.id`('appa','eomma','deahyun','jiwon')는 **auth.users.id (uuid)**로 교체. 표시명만 `display_name`으로 유지.

## 3. RLS 정책 (보안의 핵심 방어선)

셀프호스팅이어도 RLS는 PostgreSQL 표준 기능이라 그대로 동작. 모든 도메인 테이블에 동일 패턴 적용.

```sql
ALTER TABLE transactions ENABLE ROW LEVEL SECURITY;

-- 헬퍼: 현재 사용자가 속한 가족 목록
CREATE OR REPLACE FUNCTION my_households() RETURNS SETOF uuid
  LANGUAGE sql STABLE SECURITY DEFINER AS $$
  SELECT household_id FROM household_members WHERE user_id = auth.uid()
$$;

-- SELECT: 같은 가족만
CREATE POLICY tx_select ON transactions FOR SELECT
  USING (household_id IN (SELECT my_households()));

-- INSERT: 같은 가족이며 created_by는 본인 강제
CREATE POLICY tx_insert ON transactions FOR INSERT
  WITH CHECK (
    household_id IN (SELECT my_households())
    AND created_by = auth.uid()
  );

-- UPDATE/DELETE: 같은 가족 (2단계 권한이므로 모든 멤버 가능)
CREATE POLICY tx_update ON transactions FOR UPDATE
  USING (household_id IN (SELECT my_households()))
  WITH CHECK (household_id IN (SELECT my_households()));

CREATE POLICY tx_delete ON transactions FOR DELETE
  USING (household_id IN (SELECT my_households()));
```

`households`는 별도 — 본인 가족만 SELECT, owner만 UPDATE/DELETE.
`household_members`는 RLS로 모든 직접 변경 DENY → Edge Function의 서비스 키로만 변경 (사용자가 본인을 다른 가족에 끼워넣는 공격 방지).

## 4. 인증 흐름 (매직링크 단일)

### 4.1 로그인 / 회원가입 (구분 없음)

```
1. 사용자가 이메일 입력 → "로그인 / 가입 링크 받기" 클릭
2. 클라이언트: supabase.auth.signInWithOtp({ email, options: { emailRedirectTo: '/auth/callback' } })
3. GoTrue가 이메일로 6자리 OTP + 매직 링크 발송
4. 사용자 둘 중 하나로 인증:
   - 메일의 매직 링크 클릭 → 자동 로그인
   - 앱 화면에 6자리 코드 입력 → 자동 로그인
5. 첫 로그인이면 /onboarding으로 → "가족 만들기" or "초대 받았어요"
```

부모님(50~60대) 진입 흐름:
- 처음 한 번: 본인 이메일 입력 → 코드 6자리 → 자녀가 초대 토큰 전달 → 가족 합류
- 이후: 30일 리프레시 토큰으로 자동 로그인 유지

### 4.2 이메일 발송

| 단계 | 발송 수단 | 비고 |
|---|---|---|
| **초기** | GoTrue 내장 SMTP → 자택 회선의 SMTP 직접 발송 | 메이저 메일 서버 차단 가능성 있음 |
| **권장** | [Resend](https://resend.com) Free 플랜 (월 3,000건) | 1인 운영, 도메인 검증만으로 사용 가능 |
| 대안 | AWS SES, SendGrid Free | 가입 절차 더 복잡 |

→ **Resend 권장**. 가족 4인이면 평생 무료 한도 안에서 운영 가능. SPF/DKIM/DMARC를 본인 도메인에 설정 (가입 안내 따라 5분).

### 4.3 세션 정책

| 항목 | 값 |
|---|---|
| 액세스 토큰 수명 | 1시간 |
| 리프레시 토큰 수명 | 30일 (기본) / 24시간 ("이 기기 기억하기" off) |
| 저장 위치 | **httpOnly + Secure 쿠키** (LocalStorage 금지) |
| SameSite | `Lax` |
| 자동 갱신 | Supabase 클라이언트가 처리 |
| OTP 수명 | 5분 (GoTrue 기본) |
| OTP 재발송 제한 | 60초 (rate limit) |

### 4.4 회원 탈퇴

Edge Function `/account/delete`:
1. 해당 user_id의 household_members 모두 삭제 (cascade로 본인 거래의 created_by → NULL)
2. 본인이 마지막 owner인 가족이면 가족 자체 삭제 확인 모달 → cascade
3. auth.users 삭제
4. audit_log는 user_id를 NULL로 익명화 (30일 후 자동 정리 cron)

## 5. 초대 / 가족 관리

### 5.1 가족 만들기 (회원가입 직후)
- 첫 로그인 시 가족이 없으면 `/onboarding`으로 강제
- "새 가족 만들기" → households INSERT + household_members INSERT (role='owner')
- "초대받았어요" → 토큰 입력 화면

### 5.2 초대 보내기 (owner만)
```
POST /functions/v1/create-invite
  body: { household_id, email }
→ Edge Function:
   1. auth.uid()가 household의 owner인지 검증
   2. 토큰 32 bytes 생성
   3. invitations INSERT (token_hash = sha256(token), expires_at = now() + 24h)
   4. Resend로 초대 이메일 발송 (제목: "○○님이 가족 가계부에 초대했어요")
      메일 본문에 URL: https://gagyebu.yourdomain.com/invite?t=<raw_token>
   5. 응답: { invitation_id }
```

### 5.3 초대 수락
```
사용자가 메일 링크 클릭 → /invite?t=<raw_token>
→ 미로그인이면 매직링크 로그인 (이메일 입력 → OTP) 후 같은 URL로 복귀
→ POST /functions/v1/accept-invite { token }
→ Edge Function:
   1. hash(token) 조회
   2. expires_at > now() AND consumed_at IS NULL AND revoked_at IS NULL
   3. household_members INSERT (role='member')
   4. invitations UPDATE: consumed_at, consumed_by
   5. audit_log INSERT
```

### 5.4 멤버 추방 (owner만)
- Edge Function `/remove-member`로만 실행, audit_log 기록
- 본인의 거래 데이터는 가족에 남김 (created_by는 그대로, 닉네임만 "탈퇴 멤버"로 변경)

## 6. 위협 모델 & 완화

| # | 위협 | 완화 |
|---|---|---|
| 1 | **다른 가족 데이터 조회** | RLS 정책 (모든 테이블) + CI 통합 테스트 (다른 user의 JWT로 SELECT 시 0건) |
| 2 | XSS via 거래 메모/제목 | React 기본 escaping + CSP `script-src 'self'` + dangerouslySetInnerHTML 금지 |

| 9 | 초대 링크 재사용/유출 | token_hash + 24h CHECK + consumed_at + revoked_at + email rate limit |
| 10 | 의존성 취약점 | Dependabot + `npm audit --production` CI 게이트 + Docker 이미지 월 1회 `docker compose pull` |
| 11 | 시크릿 유출 | 서비스 키는 `.env` (chmod 600, git 무시). 클라이언트는 anon key + RLS만 |
| 12 | 로그 민감정보 | 금액·이름은 로그 금지, IP는 hash, audit_log diff는 PII 컬럼 마스킹 |
| 13 | **백업 유출** | `pg_dump` → age 또는 GPG 암호화 → 외장 HDD + Backblaze B2 이중화 |
| 14 | **포트 노출 / 스캔** | Cloudflare Tunnel outbound만 사용 — 라우터에서 어떤 포트도 인바운드 허용 안 함 |

| 20 | 로그 인젝션 | audit_log diff는 jsonb (구조화 저장, 임의 텍스트 아님) |

- 비밀번호·키 회전: 6개월마다 JWT_SECRET 회전 → 전 사용자 재로그인 필요

### 7.3 호스트 OS 보안 (미니 PC, Ubuntu Server 24.04 LTS 권장)
```
- 자동 보안 업데이트: unattended-upgrades 활성
- SSH: 비밀번호 로그인 비활성, ed25519 키만, Tailscale 네트워크로만 접근
- 방화벽 UFW: outbound 전부 허용, inbound는 SSH(Tailscale에서만) 외 전부 차단
- fail2ban: 혹시 모를 brute force 대비
- 비루트 사용자로 docker 실행 (rootless docker 또는 docker socket 권한)
- Docker 이미지: 공식 supabase 이미지만, latest 태그 금지 (버전 고정)
```

### 7.4 CI 게이트 (GitHub Actions)
```yaml
- npm run typecheck
- npm run lint
- npm audit --production --audit-level=high
- 통합 테스트: 다른 가족 user의 JWT로 SELECT/UPDATE/DELETE 시도 → 모두 0건 / denied
- Lighthouse Accessibility ≥ 95
- 스키마 마이그레이션 dry-run (Supabase CLI)
```

### 7.5 모니터링
- **Supabase Studio**: Auth 실패율, RLS 거부, slow query — Tailscale로만 접근
- **Uptime Kuma** (셀프호스팅 무료): 미니 PC 가동 + Cloudflare Tunnel 헬스체크, 다운 시 카카오톡/이메일 알림
- **Sentry** (클라이언트): JS 에러 + PII 자동 스크럽 (`beforeSend` 훅으로 amount/title 마스킹)
- **Resend 대시보드**: 메일 발송 실패/반송 모니터링

### 7.6 백업 / 복구
- **매일 03:00 cron**: `pg_dump --format=custom | age --encrypt -p > /backups/$(date +%Y%m%d).pgdump.age`
- **외장 HDD**: 매일 자동 복사 (rsync) — 화재·도난 1차 방어
- **클라우드 이중화**: Backblaze B2 무료 10GB → 매주 1회 업로드 (가족 4인 1년치 압축하면 100MB 미만)
- **복구 리허설**: 분기 1회, 별도 docker-compose에 복원해 데이터 일치 확인
- **보존 정책**: 일별 7개, 주별 4개, 월별 12개 (Grandfather-Father-Son)

## 8. PIPA (개인정보보호법) 준수

자택 미니 PC에 자체 보관 → **위탁업체 없음, 국외 이전 없음** → 사양 단순화.

| 항목 | 처리 |
|---|---|
| 처리방침 동의 | 회원가입 시 필수. 동의 시각 기록 |
| 수집 항목 | 필수: 이메일, 닉네임 / 선택: 없음 |
| 보유 기간 | 회원 탈퇴 후 즉시 삭제 (audit_log는 user_id NULL로 익명화 후 1년 보존 → 자동 cron 정리) |
| **위탁 업체** | **없음** (자택 보관) — 단, 메일 발송 Resend는 위탁 처리방침에 명시 (이메일 주소만 일시 위탁) |
| **데이터 위치** | **자택 미니 PC (한국)** — 국외 이전 없음 |
| 권리 행사 | 본인 데이터 다운로드(JSON) / 정정 / 삭제 — 설정 화면 셀프 서비스 |
| 책임자 정보 | 처리방침 페이지에 본인 이메일 명시 |

가족 4인 규모는 PIPA 신고 대상이 아니지만, **누가 봐도 정상적 운영**임을 보이는 형식이 보안 의식 측면에서 의미가 있습니다.

## 9. LocalStorage → 클라우드 마이그레이션

기존 Zustand persist 데이터가 있는 상태에서 첫 로그인 시:

```
1. 클라이언트가 LocalStorage에서 'gagyebu-transactions', 'gagyebu-members' 키 확인
2. 데이터 있고 가족이 막 만들어진 상태면 "기존 데이터 가져올까요?" 모달
3. 사용자 동의 → POST /functions/v1/import-local
   payload: { household_id, transactions: [...], budgets: [...] }
4. Edge Function이 다음 검증 후 일괄 INSERT:
   - 모든 항목의 amount > 0, date 형식, kind enum
   - 거래 ≤ 10,000건 (비용 폭탄 방지)
   - household_id가 호출자 본인 소속
5. 성공 시 LocalStorage에 'gagyebu-migrated-at' 마커 저장 + 원본 키 삭제
6. 이후 모든 거래는 서버가 source of truth
```

오프라인 처리는 이번 M5 스코프 밖 (M6 이후 PWA + IndexedDB 큐로 별도 진행).

## 12. 결정 보류 (M5 진행 시 재논의)

- **2FA 도입 시점**: 매직링크는 이메일 보안에 의존. 가족 누군가 이메일 계정이 탈취되면 가계부도 접근 가능. 6개월 운영 후 TOTP 추가 검토 (Supabase 내장 MFA 지원).
- **이메일 발송 전환**: 초기 Resend Free → 월 3,000건 초과 시 SES 또는 Resend 유료. 가족 4인은 평생 무료 한도 안.
- **고가용성**: 정전·인터넷 끊김 시 가족 접근 불가. 6개월 운영해서 다운타임 누적이 견딜 만한지 확인 → 견딜 수 없으면 (a) UPS + 4G 백업 회선 / (b) 클라우드 이중화 검토.
- **자녀 권한 분리**: 사용자 의견 받아 필요하면 `child` 역할 추가 (현재는 2단계 owner/member).
- **모바일 PWA**: M6 단계에서 IndexedDB 캐시 + 오프라인 큐 추가.
- **거래 첨부 (영수증 사진)**: M7. Supabase Storage 셀프호스팅 또는 가족 사진 폴더 마운트.
